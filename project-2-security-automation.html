BACKGROUND

01 기획 배경 — "왜 자동화가 필요했나"

기업의 디지털 전환과 서비스 운영 환경의 복잡도가 높아지면서,
이를 뒷받침하는 리눅스 서버·고객 데이터 DB 등 자산 보호의 중요도가 크게 높아졌습니다.

그런데 실제 현장에서는 리눅스/DB 보안 점검 항목 수가 많고 자산이 늘수록
수동 점검 → 결과 취합 → 보고서 작성 → 조치 → 이행 확인이 분절되어 운영 병목이 발생하고 있었습니다.
그 결과 시간 과다, 인적 오류, 조치 지연, 전체 가시성 부족이 누적되는 구조였습니다.

"점검만 잘하는 도구"가 아니라 컨설팅 실무 흐름 전체(점검 ~ 산출물 ~ 이력)까지
운영 가능한 자동화 체계를 만드는 것을 목표로 설정했습니다.

프로젝트 범위

Rocky Linux(9.7 / 10.2) 및 DB(MySQL 8.0.44 / PostgreSQL 16.11) 환경을 대상으로,
2026 KISA 주요정보통신기반시설 기술적 취약점 분석·평가 방법 상세가이드 기반 보안 컨설팅 자동화 플랫폼을 개발했습니다.

SCRIPT DESIGN

04 점검 · 조치 스크립트 설계 원칙

check_U00.sh(점검) / fix_U00.sh(조치) 형태로 표준화했으며, 다음 4가지 원칙을 전 스크립트에 일괄 적용했습니다.

1. 재현 가능성 — command를 통해 언제든 동일 검증 가능, 장애·감사·이슈 발생 시 당시 근거를 동일 명령으로 재검증
2. 증거 중심 — REASON_LINE(양호/취약 근거), DETAIL_CONTENT(현재 설정값)를 명시해 기준 변경 시에도 재평가 가능
3. 예외/실패 방어 — 파일 없음, 권한 부족, 서비스 미사용, 명령 실패 등을 분기 처리해 장애 환경에서도 안정적으로 결과 도출
4. 연동 안정성 — JSON 표준 출력으로 수집·저장·집계 시스템에 바로 적용, 줄바꿈/따옴표 등 JSON 형식 오류 방지


조치는 "명령 실행 성공"이 아니라 조치 후 재점검 결과를 기준으로 성공 여부를 판단하도록 설계했고,
변경 범위를 최소화·표준화해 운영 환경 변동성을 줄였습니다.

Ansible 실행 전에는 .env 파일 존재를 검증하고 fail로 즉시 실패시켜 오작동을 막았습니다.

RESULTS

07 성과 및 자체 평가

1. 대규모 자산 병렬 점검 KISA 점검 항목을 서버 100대+에 병렬 적용할 수 있도록 자동화 구조를 확장
2. 점검 → 조치 → 재점검 확보 조치 성공 여부를 명령 실행이 아닌 재점검 결과로 판정하고, 승인 기반 선택 조치로 운영 리스크를 통제
3. 이력 기반 가시성 단발 점검이 아닌 보안 점수 추세를 History 차트로 제공해 기업 전체 상태를 한 화면에서 확인
4. 컨설팅 결과 보고서 자동 생성 표지, 요약, 항목 요약, 서버별 상세 시트로 구성된 엑셀 보고서 자동 생성
5. 4계층 보안 구조 설계 내/외부망 분리(CIDR), JWT 인증, RBAC 접근제어, 민감정보 Fernet/PBKDF2 암호화 반영
